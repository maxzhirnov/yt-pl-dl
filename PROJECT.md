# yt-pl-dl

## Цель

Сервис должен:

1. Проверять указанный YouTube playlist.
2. Определять, появилось ли новое видео.
3. Скачивать новые видео локально.
4. Переносить и синхронизировать скачанные файлы в папку на Synology DSM в локальной сети.
5. Дальше Plex подхватывает файлы уже со стороны Synology.

Исходный playlist:

- `https://youtube.com/playlist?list=PLG0cTqREUrX-BE3_hOOu1O94OP1ZI6vOw`

## Рекомендуемый стек

Основной язык: Python.

Почему Python здесь подходит:

- есть зрелые инструменты для YouTube, в первую очередь `yt-dlp`;
- просто запускать как cron/systemd service;
- удобно хранить состояние в `sqlite` или JSON;
- легко добавить логирование, retry и конфиг через `.env`.

## Базовая архитектура

### 1. Playlist checker

Скрипт получает список видео из playlist и сравнивает его с локально сохраненным состоянием.

Предпочтительный вариант:

- использовать `yt-dlp --flat-playlist` или Python API `yt_dlp` для получения metadata без скачивания всего видео;
- хранить список уже обработанных `video_id`.

### 2. Downloader

Если найден новый `video_id`, сервис скачивает видео локально в рабочую директорию.

Рекомендации:

- скачивать через `yt-dlp`;
- имя файла делать стабильным и безопасным, например `%(upload_date)s - %(title)s [%(id)s].%(ext)s`;
- хранить видео и metadata рядом.

### 3. State storage

Для начала достаточно одного из вариантов:

- `state.db` на SQLite;
- или `state.json`, если нужен самый простой старт.

Я рекомендую `sqlite`, потому что дальше почти наверняка захочется:

- видеть статус загрузки;
- хранить время последней проверки;
- фиксировать ошибки синхронизации;
- не ломаться при частичных перезапусках.

### 4. Sync to Synology

Лучший практический вариант зависит от того, как у тебя доступен NAS.

Приоритет такой:

1. Если папка Synology смонтирована локально, писать сразу в mounted path.
2. Если есть SSH на Synology, синхронизировать через `rsync`.
3. Если есть только SMB, можно либо писать в mounted share, либо отдельно настраивать копирование через сетевой mount.

Для первой версии я рекомендую `rsync`, если SSH доступен:

- надежная дозагрузка;
- нормальная синхронизация;
- легко повторять после ошибок.

### 5. Scheduler

Запускать сервис можно так:

- `cron` для простого периодического запуска;
- `systemd timer`, если хочется более аккуратный Linux deployment;
- ручной loop внутри контейнера или процесса, если позже завернем в Docker.

Для MVP самый простой вариант: периодический запуск раз в 10-30 минут.

## MVP

Первая рабочая версия должна уметь:

1. Проверить playlist.
2. Найти новые видео.
3. Скачать новые видео локально.
4. Скопировать или синхронизировать их в папку Synology.
5. Не скачивать одно и то же повторно.

## Конфиг

Пример будущих переменных окружения:

```env
YT_PLAYLIST_URL=https://youtube.com/playlist?list=PLG0cTqREUrX-BE3_hOOu1O94OP1ZI6vOw
LOCAL_DOWNLOAD_DIR=./data/downloads
STATE_DB_PATH=./data/state.db
SYNC_MODE=rsync
SYNC_TARGET=/path/to/mounted/or/remote/folder
RSYNC_HOST=192.168.x.x
RSYNC_USER=your-user
RSYNC_REMOTE_PATH=/volume1/media/youtube
CHECK_INTERVAL_MINUTES=15
```

## Решения, которые зафиксированы сейчас

- Язык: Python.
- Получение playlist metadata: `yt-dlp`.
- Состояние: `sqlite`.
- Первая стратегия синхронизации: `rsync`, либо запись в уже смонтированную папку Synology.
- Формат сервиса: сначала CLI-утилита, потом при необходимости daemon/container.

## Вопросы перед реализацией

Нужно уточнить:

1. Где будет работать сервис: на Mac, Linux mini-PC, Docker-host, NAS?
2. Есть ли SSH-доступ на Synology?
3. Папка DSM уже смонтирована локально или нет?
4. Нужно ли скачивать видео в максимальном качестве или есть ограничения по размеру?
5. Нужны ли уведомления при ошибке или новом видео?
